# Feature Specification: AnyDrop 移动端 App（MAUI Blazor Hybrid）

**Feature Branch**: `006-maui-mobile-app`
**Created**: 2026-04-29
**Status**: Draft

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 首次配置与登录（P1）

用户首次安装 AnyDrop App 后，需要输入自托管服务端地址，然后完成账号登录（或在服务端尚未初始化时完成首次账号创建），之后进入主界面开始使用。

**Why this priority**: 这是所有功能的前提条件。没有服务端连接和有效认证，App 无任何可用功能。

**Independent Test**: 在一台全新设备上安装 App，输入服务端 URL，完成登录后能看到主界面，即为完整可验证的 MVP 路径。

**Acceptance Scenarios**:

1. **Given** 用户首次启动 App（无已保存服务端地址），**When** 进入引导页并输入有效服务端 URL，**Then** App 保存该地址并自动跳转至登录页或首次初始化页（依服务端 `/api/v1/auth/setup-status` 响应决定）。
2. **Given** 服务端返回 `requiresSetup: true`，**When** 用户在初始化页填写昵称与密码并提交，**Then** 账号创建成功并自动登录，跳转至主界面。
3. **Given** 服务端已有账号（`requiresSetup: false`），**When** 用户输入正确密码，**Then** 登录成功，JWT 令牌保存至安全存储，跳转主界面。
4. **Given** 用户输入了错误密码，**When** 点击登录，**Then** 显示友好错误提示，保持在登录页。
5. **Given** 输入了无法访问的服务端地址，**When** App 尝试连接，**Then** 显示"无法连接到服务端"提示，允许用户重新输入地址。

### User Story 2 — 消息收发（主聊天界面）（P1）

用户可在主界面查看主题侧边栏、切换主题，并在当前主题中发送文本消息、上传文件（图片、视频、任意文件），同时实时接收其他端推送的新消息。

**Why this priority**: 这是 AnyDrop 的核心功能，跨设备内容共享的直接承载界面。

**Independent Test**: 在主界面选择一个主题后发送一条文本消息，能看到该消息出现在消息列表，即为独立可测试的核心交互。

**Acceptance Scenarios**:

1. **Given** 用户已登录并进入主界面，**When** 页面加载，**Then** 主题列表（含置顶、普通）从服务端加载并显示在侧边栏（或主页列表）。
2. **Given** 用户点击某主题，**When** 主题被选中，**Then** 该主题的历史消息按时间顺序展示（最新消息在底部），支持向上滚动加载更早消息（分页）。
3. **Given** 消息列表已加载，**When** 用户向上滚动到顶部，**Then** 自动加载更早的一批消息并插入列表顶部。
4. **Given** 用户在输入框输入文本并点击发送，**When** 发送成功，**Then** 消息立即出现在列表，服务端通过 SignalR 同步推送到其他已连接客户端。
5. **Given** 用户点击"上传文件"按钮，**When** 选择图片/视频/文件，**Then** 文件以 multipart 方式上传到服务端，上传进度可见，成功后消息出现在列表。
6. **Given** SignalR 连接正常，**When** 其他设备发送了新消息，**Then** 本端实时收到消息并追加至列表底部，不需要手动刷新。
7. **Given** 消息列表中包含图片消息，**When** 用户点击图片，**Then** 进入全屏图片预览模式，支持手势缩放。
8. **Given** 消息列表中包含链接消息，**When** 消息卡片渲染，**Then** 展示链接标题、描述等 OGP 预览信息（如服务端已提取）。

### User Story 3 — 主题管理（P2）

用户可创建、重命名、删除主题，设置主题图标，置顶或归档主题，并调整主题排序。

**Why this priority**: 主题是内容组织的核心结构，但在有默认主题的情况下用户可先收发消息再管理主题。

**Independent Test**: 从空主题列表创建一个新主题并发送一条消息，即为完整独立路径。

**Acceptance Scenarios**:

1. **Given** 用户在主界面点击"新建主题"，**When** 输入主题名称并确认，**Then** 主题在侧边栏中出现，服务端已保存。
2. **Given** 用户长按（或通过菜单打开）已有主题，**When** 选择"重命名"，**Then** 可修改主题名称，保存后立即更新。
3. **Given** 用户打开主题设置，**When** 选择"更改图标"并选取 emoji 或图标，**Then** 主题图标更新，列表中即时反映。
4. **Given** 用户拖拽主题到新位置，**When** 松手，**Then** 排序保存到服务端，刷新后顺序一致。
5. **Given** 用户对主题选择"置顶"，**When** 确认，**Then** 该主题移至列表顶部，带置顶标记。
6. **Given** 用户对主题选择"归档"，**When** 确认，**Then** 该主题从活跃列表移出，可在"已归档"区域查看。
7. **Given** 用户对主题选择"删除"，**When** 二次确认后，**Then** 主题及其所有消息从服务端删除，本地列表同步移除。

---

### User Story 4 — 搜索（P2）

用户可在搜索页通过文本关键词、日期、内容类型等条件查找历史消息。

**Why this priority**: 搜索是辅助功能，主收发流程不依赖它，但对历史内容检索有重要价值。

**Independent Test**: 在搜索页输入一个关键词，结果列表中展示匹配消息，即为独立可验证路径。

**Acceptance Scenarios**:

1. **Given** 用户进入搜索页并输入关键词，**When** 触发搜索（实时或点击搜索按钮），**Then** 显示包含该关键词的消息列表，支持分页加载更多结果。
2. **Given** 用户在搜索页选择"按日期查找"并在日历上点击高亮日期，**When** 确认，**Then** 列出该日所有消息。
3. **Given** 用户在搜索页选择"按类型浏览"，**When** 选择"图片"，**Then** 以缩略图网格展示所有图片消息。
4. **Given** 搜索无结果，**When** 列表渲染，**Then** 显示"未找到相关内容"空状态提示。

### User Story 5 — 设置（P2）

用户可在设置页修改昵称、密码，调整安全设置（自动清理、阅后即焚时长、链接预览、语言），手动触发旧消息清理，并切换应用主题（深色/浅色）。

**Why this priority**: 设置影响整体体验，但不阻塞核心收发功能，优先级低于消息交互。

**Independent Test**: 进入设置页修改昵称并保存，主界面顶部显示更新后的昵称，即为独立路径。

**Acceptance Scenarios**:

1. **Given** 用户在设置页"个人资料"区域修改昵称，**When** 保存，**Then** 昵称更新成功，当前会话中显示新昵称。
2. **Given** 用户在"安全设置"中修改密码，**When** 填写当前密码和新密码并确认，**Then** 密码更新成功，下次登录使用新密码。
3. **Given** 用户在设置页选择语言，**When** 切换为另一语言，**Then** App 界面文字立即切换（中文 ↔ 英文）。
4. **Given** 用户在设置页切换主题，**When** 选择深色模式，**Then** App 整体色调切换为深色。
5. **Given** 用户点击"手动清理"并输入保留月数，**When** 确认，**Then** 早于指定时间的消息从服务端删除，操作结果通过提示反馈。
6. **Given** 用户在设置页可见"服务端地址"入口，**When** 点击并修改 URL，**Then** App 使用新地址重新连接，需重新登录。
7. **Given** 用户点击"退出登录"，**When** 确认，**Then** 本地 JWT 令牌清除，跳转至登录页。

---

### User Story 6 — 接收外部分享内容（P3）

用户可从其他 App（如浏览器、相册、文件管理器）通过系统"分享"功能将文字、图片、文件直接分享到 AnyDrop App 中，选择目标主题后完成发送。

**Why this priority**: 这是移动端特有的附加体验增强，核心功能不依赖它。

**Independent Test**: 在系统相册中选择一张图片，通过系统分享菜单选择 AnyDrop，选择目标主题后图片发送成功，即为完整路径。

**Acceptance Scenarios**:

1. **Given** 用户在其他 App 中选择"分享"并选中 AnyDrop，**When** AnyDrop 被唤起，**Then** 显示主题选择界面，内容预填充（文字/文件预览）。
2. **Given** 用户选择目标主题并点击"发送"，**When** 发送成功，**Then** 内容出现在该主题的消息列表，操作界面关闭或返回上一 App。
3. **Given** 用户分享时 App 未登录，**When** AnyDrop 被唤起，**Then** 跳转至登录页，登录后继续分享流程。

---

### User Story 7 — 新消息本地通知（P3）

用户在 App 退至后台或锁屏期间，收到新消息时通过系统推送通知提醒。

**Why this priority**: 提升用户感知度的增强功能，非核心路径。

**Independent Test**: 将 App 切换到后台，从另一设备发送一条消息，本机出现系统通知，即为可测路径。

**Acceptance Scenarios**:

1. **Given** App 在后台且 SignalR 连接保持活跃，**When** 新消息到达，**Then** 系统通知栏出现消息通知（含主题名和消息摘要）。
2. **Given** 用户点击通知，**When** App 被唤起，**Then** 直接跳转至对应主题的消息界面。
3. **Given** 用户在设置中关闭通知权限，**When** 新消息到达，**Then** 不再产生系统通知。

---

### Edge Cases

- 服务端地址填写格式错误（如缺少协议前缀），App 给出格式校验提示而非发起无效请求。
- JWT 令牌过期时，App 自动清除令牌并跳转至登录页，不展示原始 401 错误信息。
- 文件上传超出服务端限制时，给出文件大小提示，不中断整体 App 运行。
- SignalR 断线后，App 自动尝试重连（指数退避），重连期间显示非阻断式连接状态指示器。
- 无网络或服务端不可达时，已加载的消息仍可浏览（只读），并显示离线状态横幅。
- 切换主题时的未发送草稿（输入框内容）在本次会话内保留，不跨会话持久化。
- 在图片全屏预览状态下，系统返回手势/按钮退出预览而非退出 App。
- 从外部分享进入但网络不可用时，给出提示，分享内容可短暂缓存待用户恢复连接后重试。

## Requirements *(mandatory)*

### Functional Requirements

#### 连接与认证

- **FR-001**: App 首次启动时 MUST 展示服务端 URL 配置页，用户输入并验证后方可继续。
- **FR-002**: App MUST 持久化服务端地址至本地安全存储，重启后无需重新输入。
- **FR-003**: App MUST 在启动时调用 `/api/v1/auth/setup-status` 判断是否需要初始化，并路由至对应页面（Setup 或 Login）。
- **FR-004**: App MUST 将登录后的 JWT 令牌存储至平台安全存储（Android Keystore / iOS Keychain），令牌不得以明文写入普通文件或 Preferences。
- **FR-005**: App MUST 在 JWT 令牌检测到过期或收到 401 响应时，自动清除本地令牌并导航至登录页。
- **FR-006**: App MUST 支持退出登录，退出后清除本地令牌并调用服务端 `/api/v1/auth/logout`。

#### 主题管理

- **FR-007**: App MUST 从服务端加载活跃主题列表，并区分展示置顶主题与普通主题。
- **FR-008**: App MUST 支持创建主题（名称、可选图标）。
- **FR-009**: App MUST 支持重命名主题、更改主题图标。
- **FR-010**: App MUST 支持置顶、归档主题。
- **FR-011**: App MUST 支持拖拽调整主题排序，排序结果持久化到服务端。
- **FR-012**: App MUST 支持删除主题（需二次确认），删除后本地列表同步更新。
- **FR-013**: App MUST 提供已归档主题的独立查看入口。

#### 消息收发

- **FR-014**: App MUST 展示当前主题的历史消息列表，按发送时间升序排列（最新在底部）。
- **FR-015**: App MUST 支持向上滚动触发分页加载，每次以 `before` 游标向服务端请求更早消息，加载到最早消息后不再触发。
- **FR-016**: App MUST 支持发送文本消息（调用 `/api/v1/share-items/text`）。
- **FR-017**: App MUST 支持上传文件，包含图片、视频、任意格式文件（multipart 上传），上传期间显示进度指示。
- **FR-018**: App MUST 正确渲染五种消息类型：文本（Text）、图片（Image）、视频（Video）、文件（File）、链接（Link）。
- **FR-019**: App MUST 通过 SignalR 连接到服务端 Hub，实时接收新消息推送（`TopicsUpdated` 事件），无需手动刷新。
- **FR-020**: App MUST 在 SignalR 断连后自动重连，重连期间以非阻断方式提示用户（如状态条），不影响已加载内容的浏览。
- **FR-021**: App MUST 支持点击图片消息进入全屏预览，全屏预览支持双指缩放与单指平移。
- **FR-022**: App MUST 支持下载文件消息（调用 `/api/v1/share-items/{id}/file?download=true`）。

#### 搜索

- **FR-023**: App MUST 提供独立搜索页，支持在全部主题范围内按文本关键词搜索消息（调用 `GET /api/v1/topics/{id}/messages/search`）。
- **FR-024**: App MUST 支持在搜索页选择日期查看当日消息（`/messages/by-date`）。
- **FR-025**: App MUST 支持在搜索页通过日历控件查看哪些日期有消息（`/active-dates`），并高亮显示。
- **FR-026**: App MUST 支持按内容类型筛选消息（`/messages/by-type`），类型包含：文本、图片、视频、文件、链接。
- **FR-027**: 搜索结果 MUST 支持分页加载更多，到达末尾时给出提示。

#### 设置

- **FR-028**: App MUST 提供设置页，包含以下子功能：昵称修改（`PUT /api/v1/settings/profile`）、密码修改（`PUT /api/v1/settings/password`）、安全设置查看与更新（`GET/PUT /api/v1/settings/security`）。
- **FR-029**: App MUST 支持手动触发旧消息清理（`DELETE /api/v1/share-items/cleanup`），用户指定保留月数并在操作前二次确认。
- **FR-030**: App MUST 支持语言切换（中文 / 英文），切换后 UI 立即生效，选择持久化。
- **FR-031**: App MUST 支持深色 / 浅色主题切换，选择持久化至本地。
- **FR-032**: App MUST 在设置页提供"修改服务端地址"入口，修改后需重新认证。

#### 移动端特有

- **FR-033**: App MUST 在无网络或服务端不可达时显示离线提示横幅（非阻断式），已加载内容保持可浏览。
- **FR-034**: App MUST 实现 Android & iOS 平台的"接收外部分享"能力（Share Extension / Intent Filter），接收文字、图片、文件，唤起后展示主题选择界面。
- **FR-035**: App SHOULD 在新消息到达且 App 处于后台时触发本地通知（可选，依平台通知权限）。
- **FR-036**: App MUST 提供底部导航栏（主页、搜索、设置），在三个主要页面间快速切换。

### Key Entities

- **ServerConfig**: 用户配置的服务端 URL，本地持久化，唯一。
- **AuthSession**: JWT 令牌、过期时间、用户昵称，存储于平台安全存储。
- **Topic**: 主题 ID、名称、图标、置顶标志、归档标志、排序序号。
- **ShareItem（消息）**: 消息 ID、内容类型（Text / Image / Video / File / Link）、内容（文本或文件路径/URL）、文件名、文件大小、MIME 类型、链接标题/描述、创建时间、过期时间（阅后即焚）、所属主题 ID。
- **SecuritySettings**: 自动获取链接预览开关、阅后即焚时长（分钟）、语言设置、自动清理开关、自动清理保留月数。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从首次启动到发出第一条消息的完整流程（配置服务端 → 登录 → 发送消息）可在 3 分钟内完成。
- **SC-002**: 消息列表在常规网络环境下加载首屏内容的等待时长不超过 2 秒（含认证完成后的主题与消息双重请求）。
- **SC-003**: 从另一设备发送消息到本端通过 SignalR 收到并展示，端到端延迟在正常局域网条件下不超过 1 秒。
- **SC-004**: 文件上传成功率在稳定网络下达到 99% 以上（不含用户主动取消）。
- **SC-005**: App 在 Android 和 iOS 两个平台均可正常运行，所有 P1/P2 用户故事的验收场景通过率达 100%。
- **SC-006**: App 冷启动（从图标点击到主界面可交互）时间在中端设备上不超过 4 秒。
- **SC-007**: 用户可在搜索页通过关键词在 5 秒内定位到目标历史消息（不含服务端查询超时情况）。
- **SC-008**: App 在连接断开后，SignalR 自动重连成功率（连续重试 5 次内）不低于 95%（前提是网络已恢复）。

## Assumptions

- 服务端已按规范部署并通过 HTTPS 或局域网 HTTP 提供服务；App 不对服务端部署方式做任何假设，仅需用户提供可访问的 URL。
- AnyDrop 为单用户系统（每个服务端实例仅一个账号），App 不需要多账号切换功能。
- App 不在本地缓存消息数据库；所有消息数据来自服务端 API，本地仅缓存当次会话内已加载的消息以支持滚动浏览。
- 图片/视频全屏预览仅针对服务端存储的媒体文件（通过 `/api/v1/share-items/{id}/file` 获取），不支持离线缓存播放。
- 外部分享功能（Share Extension）在目标主题未选择时不自动发送，用户必须主动确认。
- 本地通知功能（P3）依赖用户授予通知权限；若权限被拒绝，App 不反复请求，仅在设置页提示用户手动开启。
- 语言支持范围为中文（简体）和英文，其他语言不在本期范围内。
- App 使用 Blazor Hybrid 架构，页面 UI 通过 Blazor 组件实现，复用 Web 端已有 Tailwind CSS 样式，适配移动端触控交互（触摸目标尺寸符合平台规范）。
- 深色/浅色主题切换通过 CSS 类切换实现，跟随用户手动选择，不强制跟随系统主题（可在设置中配置）。
- 服务端 JWT 认证使用 Bearer Token，通过 HTTP Authorization Header 传递；SignalR 连接使用相同 JWT 进行认证。
