# Feature Specification: 富媒体聊天增强

**Feature Branch**: `003-rich-media-chat`  
**Created**: 2026-04-19  
**Status**: Draft  
**Input**: 在 MVP 基础上丰富聊天功能，包括主题置顶、快捷键发送、超链接解析、图片/视频异步上传与预览、历史记录懒加载、附件支持。

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 主题置顶与标题显示 (Priority: P1)

用户在右侧聊天区域顶部可看到当前主题的名称作为标题，并可通过一个"置顶"开关将该主题固定在左侧主题列表顶部，不受最新消息时间排序的影响。

**Why this priority**: 主题标题和置顶是导航核心体验，直接影响多主题管理效率，是其他功能的上下文基础。

**Independent Test**: 进入任意主题聊天界面，验证顶部标题显示正确；点击置顶按钮后刷新页面，确认该主题始终排在列表最顶部，即使其他主题有更新消息。

**Acceptance Scenarios**:

1. **Given** 用户进入某主题, **When** 查看聊天区域顶部, **Then** 显示该主题的名称作为标题
2. **Given** 用户点击置顶按钮激活置顶, **When** 查看主题列表, **Then** 该主题固定显示在列表最顶部，即使有其他主题收到新消息
3. **Given** 用户再次点击置顶按钮取消置顶, **When** 查看主题列表, **Then** 该主题回归按最新消息时间排序的正常位置
4. **Given** 多个主题均被置顶, **When** 查看主题列表, **Then** 所有置顶主题优先显示，相互之间按置顶时间排序

---

### User Story 2 - Ctrl+Enter 快捷键发送 (Priority: P1)

用户在消息输入框中按下 Ctrl+Enter 即可快速发送消息，与点击发送按钮效果相同，提升键盘操作效率。

**Why this priority**: 高频操作的快捷键支持直接改善用户体验，且实现成本低。

**Independent Test**: 在输入框中输入文本后按 Ctrl+Enter，确认消息被发送，输入框清空。

**Acceptance Scenarios**:

1. **Given** 输入框有内容, **When** 用户按下 Ctrl+Enter, **Then** 消息发送成功，输入框清空
2. **Given** 输入框为空, **When** 用户按下 Ctrl+Enter, **Then** 不发送消息，无任何副作用
3. **Given** 用户仅按 Enter（不含 Ctrl）, **When** 任何时候, **Then** 不触发发送，保持换行行为

---

### User Story 3 - 超链接解析与展示 (Priority: P2)

用户将 URL 粘贴到输入框并发送后，消息气泡中除显示原始链接文本外，还会自动展示该网页的元信息（标题和描述），帮助接收方快速理解链接内容。

**Why this priority**: 超链接解析提升内容传播效率，且实现复杂度适中。

**Independent Test**: 发送一条包含 HTTPS URL 的消息，等待解析完成，确认气泡中显示网页标题和描述。

**Acceptance Scenarios**:

1. **Given** 消息中包含有效 HTTPS URL, **When** 消息发送后, **Then** 系统在后台解析 URL 的 meta 信息，气泡中显示网页标题和描述
2. **Given** URL 无法访问或解析超时, **When** 解析失败, **Then** 仅显示原始链接，不显示错误信息
3. **Given** 消息包含多个 URL, **When** 发送后, **Then** 至少解析并展示第一个 URL 的 meta 信息
4. **Given** URL 对应的页面没有 meta 描述, **When** 解析完成, **Then** 仅显示标题，描述区域不展示

---

### User Story 4 - 图片/视频异步上传与进度展示 (Priority: P2)

用户点击多媒体按钮选择图片或视频后（或将文件拖拽至消息框），立即自动发送——无需点击发送按钮或输入任何文字。消息气泡立即出现（显示上传进度），文件在后台异步上传。上传成功后气泡显示缩略图/预览图，失败时显示明确错误提示。

**Why this priority**: 异步上传是大文件发送场景的基础体验保障，防止页面卡顿；拖拽上传进一步降低操作摩擦。

**Independent Test**: 选择一张图片（或拖拽图片到消息框），确认无需任何额外操作即自动发送，气泡立即出现并显示进度条；上传完成后确认进度条消失、缩略图显示正确。

**Acceptance Scenarios**:

1. **Given** 用户通过按钮选择图片文件或拖拽图片到消息框, **When** 选择/放置完成, **Then** 无需确认步骤，消息气泡立即出现并显示上传进度指示
2. **Given** 图片正在上传, **When** 上传完成, **Then** 气泡内显示图片缩略图，进度指示消失
3. **Given** 视频文件正在上传, **When** 上传完成, **Then** 气泡内显示视频预览帧（首帧截图），进度指示消失
4. **Given** 文件上传过程中出现错误, **When** 上传失败, **Then** 气泡显示"上传失败"提示，并提供重试按钮
5. **Given** 用户点击气泡上的重试按钮, **When** 重试触发, **Then** 原气泡状态重置为"上传中"并重新上传，不新增气泡，成功后显示内容，再次失败仍显示错误提示
5. **Given** 用户选择多个文件, **When** 确认发送, **Then** 每个文件对应独立气泡，各自显示上传进度

---

### User Story 5 - 历史记录懒加载与"回到最新"按钮 (Priority: P2)

聊天区域初始仅加载最近 20 条消息，用户向上滚动时自动加载更早的历史记录。当用户滚动离开底部时，右下角显示"回到最新消息"按钮，点击后平滑滚动到最新消息。

**Why this priority**: 懒加载直接改善含大量多媒体消息时的首屏加载速度和交互流畅度。

**Independent Test**: 在有 50+ 条消息的主题中进入聊天，确认只加载了 20 条；向上滚动到顶部，确认自动加载更早消息；滚动后确认右下角显示"回到最新"按钮，点击后跳到最新消息。

**Acceptance Scenarios**:

1. **Given** 主题有超过 20 条消息, **When** 进入聊天, **Then** 初始仅显示最近 20 条消息，加载速度明显快于全量加载
2. **Given** 用户已滚动到消息列表顶部, **When** 还有更多历史消息, **Then** 系统自动加载更早的一批消息并显示在顶部
3. **Given** 用户向上滚动离开最底部, **When** 距底部超过一定距离, **Then** 右下角出现"回到最新消息"浮动按钮
4. **Given** "回到最新消息"按钮可见, **When** 用户点击, **Then** 消息列表平滑滚动到最新一条消息，按钮消失
5. **Given** 所有历史消息已全部加载, **When** 用户滚动到最顶部, **Then** 显示"已加载全部消息"提示，不再触发加载

---

### User Story 6 - 多媒体消息点击预览与信息展示 (Priority: P3)

用户点击聊天气泡中的图片缩略图或视频预览图，弹出全屏模态框显示原图或播放视频，同时展示文件元信息并提供下载按钮。

**Why this priority**: 预览功能提升多媒体消息的可用性，但非核心发送流程。

**Independent Test**: 点击一条图片消息气泡，确认弹出模态框显示原图、文件名、大小、上传日期，并可点击下载按钮下载文件。

**Acceptance Scenarios**:

1. **Given** 用户点击图片气泡缩略图, **When** 点击事件触发, **Then** 弹出模态框显示原图，并展示文件名、文件大小、上传日期、图片分辨率
2. **Given** 用户点击视频气泡预览图, **When** 点击事件触发, **Then** 弹出模态框可播放视频，并展示文件名、文件大小、上传日期、视频时长
3. **Given** 模态框已打开, **When** 用户点击下载按钮, **Then** 文件以 Content-Disposition: attachment 方式下载
4. **Given** 模态框已打开, **When** 用户点击模态框背景或关闭按钮, **Then** 模态框关闭，返回聊天界面

---

### User Story 7 - 附件异步上传与预览 (Priority: P3)

用户通过独立附件按钮选择任意类型文件（或将文件拖拽至消息框），立即自动发送，无需额外确认。气泡显示上传进度及结果。点击附件气泡可预览文本类文件内容或查看文件信息，并可下载。

**Why this priority**: 附件功能扩展内容类型支持，但相比图片/视频场景使用频率稍低。

**Independent Test**: 选择（或拖拽）一个 PDF 文件，确认无需额外操作即自动发送，气泡立即出现上传进度；上传成功后确认显示附件图标；点击气泡，确认弹出模态框显示文件内容（PDF 情况下）和下载按钮。

**Acceptance Scenarios**:

1. **Given** 用户通过按钮选择文件或拖拽文件到消息框, **When** 选择/放置完成, **Then** 无需确认步骤，气泡立即出现显示上传进度
2. **Given** 附件上传成功, **When** 完成时, **Then** 气泡显示附件图示及文件名，进度指示消失
3. **Given** 附件上传失败, **When** 出现错误, **Then** 气泡显示"上传失败"提示
4. **Given** 用户点击 txt、pdf、md 类型附件气泡, **When** 点击事件触发, **Then** 弹出模态框显示文件内容预览
5. **Given** 用户点击非文本类型附件气泡, **When** 点击事件触发, **Then** 弹出模态框显示文件图示，不显示内容预览
6. **Given** 模态框已打开, **When** 用户点击下载按钮, **Then** 附件以 Content-Disposition: attachment 方式下载
7. **Given** 模态框已打开, **When** 展示附件信息, **Then** 显示文件名、文件大小、上传日期

---

### Edge Cases

- 用户上传超大文件（如 >100MB 视频）时，进度展示是否正确且不卡顿？
- 网络中断时，上传失败的气泡状态如何保留和展示？
- 解析超链接时，URL 包含特殊字符或重定向时能否正确处理？
- 视频缩略图生成失败时（如不支持的编解码格式或未安装 FFmpeg），气泡降级显示通用视频图标 + 文件名，点击后模态框仍可播放
- 同一主题内被多个设备并发置顶/取消置顶时，状态如何同步？
- 历史记录加载中途切换主题，是否会产生数据混淆？
- 用户通过拖拽或按钮同时选择多个文件时，各文件按顺序各自识别类型并创建独立气泡分别异步上传，顺序和状态互不干扰
- 输入框已有文字内容时，用户通过拖拽放入文件：文件立即作为独立消息发送，输入框文字内容保持不变，等待用户主动发送，两者互不干扰
- 拖拽不支持或危险类型的文件时（如 .exe），系统如何响应？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 聊天区域顶部标题栏 MUST 显示当前主题名称
- **FR-002**: 聊天区域标题栏 MUST 提供置顶切换按钮，激活后主题在侧边栏列表中始终排在未置顶主题之前
- **FR-003**: 置顶状态 MUST 持久化保存，刷新页面后仍有效
- **FR-004**: 消息输入框 MUST 支持 Ctrl+Enter 快捷键触发发送
- **FR-005**: 系统 MUST 在消息发送后识别消息文本中的 URL（HTTP 和 HTTPS 均支持），并在后台异步抓取对应网页的 title 和 description meta 标签内容；是否允许解析 HTTP 链接可通过应用设置项配置
- **FR-006**: URL meta 解析结果 MUST 在获取后实时更新对应气泡展示，无需用户刷新
- **FR-007**: 系统 MUST 提供多媒体选择按钮，支持选择图片（jpg、png、gif、webp 等）和视频（mp4、mov、webm 等）文件；同时支持将图片/视频文件拖拽到消息输入框区域触发相同行为
- **FR-007a**: 每条消息 MUST 只含单一内容类别（文字 / 图片 / 视频 / 附件），系统不支持同一条消息混合多媒体与文字内容
- **FR-007b**: 用户通过按钮或拖拽一次选择多个文件时，系统 MUST 按文件顺序依次为每个文件创建独立消息气泡，并各自识别类型（图片/视频/附件）分别处理，不限制同批次文件的类型一致性
- **FR-008**: 用户通过按钮选择或拖拽放置多媒体文件后，MUST 立即自动发送并创建消息记录显示上传进度气泡，无需用户额外点击发送或预先输入文字
- **FR-009**: 多媒体上传成功后，MUST 在气泡中显示图片缩略图或视频预览帧
- **FR-010**: 多媒体上传失败后，MUST 在气泡中显示失败提示及重试按钮；点击重试时 MUST 原地重置气泡状态为"上传中"并重新上传，不新增新气泡
- **FR-011**: 系统 MUST 在后台任务中自动生成图片缩略图和视频首帧预览图并持久化存储
- **FR-012**: 聊天历史记录 MUST 默认仅加载最近 20 条消息，向上滚动时分页异步加载更早消息
- **FR-013**: 当消息列表滚动位置不在底部时，MUST 在列表右下角显示"回到最新消息"浮动按钮
- **FR-014**: 点击"回到最新消息"按钮 MUST 将消息列表滚动到最新一条消息
- **FR-015**: 点击图片/视频气泡 MUST 弹出模态框，图片显示原图，视频可播放，并展示文件元信息和下载按钮
- **FR-016**: 系统 MUST 提供独立的附件选择按钮，支持任意文件类型的异步上传；同时支持将文件拖拽到消息输入框区域触发相同行为
- **FR-016a**: 用户通过按钮选择或拖拽放置附件文件后，MUST 立即自动发送并创建消息记录，无需额外点击发送或预先输入文字
- **FR-017**: 附件上传成功后 MUST 在气泡中显示附件图示及文件名；上传失败时显示失败提示
- **FR-018**: 点击附件气泡 MUST 弹出模态框，txt、pdf、md 等文本文件 MUST 在模态框内展示文件内容预览
- **FR-019**: 所有文件下载 MUST 使用 Content-Disposition: attachment 响应头

### Key Entities

- **Topic（主题）**: 新增 `IsPinned`（是否置顶）和 `PinnedAt`（置顶时间）字段，用于侧边栏排序
- **ShareItem（消息）**: 新增 `ContentType`（文本/超链接/图片/视频/附件）、`UploadStatus`（上传中/成功/失败）、`FilePath`（文件路径）、`ThumbnailPath`（缩略图路径）、`FileSize`（文件大小）、`OriginalFileName`（原始文件名）字段
- **LinkPreview（链接预览）**: 关联 ShareItem，存储解析到的 `Url`、`Title`、`Description`
- **MediaMetadata（媒体元数据）**: 关联 ShareItem，存储 `Width`、`Height`（分辨率）、`DurationSeconds`（视频时长）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 主题置顶后，刷新页面该主题仍在列表顶部，100% 的操作持久化生效
- **SC-002**: 用户通过 Ctrl+Enter 发送消息的成功率与点击发送按钮一致
- **SC-003**: 选择图片后，气泡在 500ms 内出现并开始显示上传进度，用户不感知页面卡顿
- **SC-004**: 聊天历史记录初始加载时间（20 条消息）相较全量加载减少 60% 以上（在 100 条消息场景下）
- **SC-005**: 超链接 meta 解析在 URL 可访问的情况下，95% 的请求在 5 秒内完成并更新气泡
- **SC-006**: 图片/视频缩略图后台生成后，页面刷新时缩略图正确显示，覆盖率达到 100% 成功上传的文件
- **SC-007**: 用户点击多媒体气泡后，模态框在 300ms 内打开并展示内容
- **SC-008**: 附件上传的失败状态 100% 在气泡中明确提示，不存在静默失败

## Clarifications

### Session 2026-04-19

- Q: 多媒体和附件是否支持拖拽上传？ → A: 支持。将文件拖拽到消息输入框区域即自动发送并上传，行为与点击按钮选择文件一致。
- Q: 一条消息是否可以同时包含文字和媒体/附件？ → A: 不支持混合。每条消息只能是单一内容类别（纯文字、纯图片、纯视频或纯附件），不存在一条消息既含多媒体/附件又含文字的情况。
- Q: 选择多媒体或附件文件后是否需要用户点击发送确认？ → A: 无需确认。一旦通过按钮选择或拖拽放置文件，立即自动发送并开始异步上传，不要求用户预先输入文字或点击发送按钮。
- Q: 输入框已有文字时拖拽文件，文字内容如何处理？ → A: 文件立即独立发送，输入框中的文字内容保持不变，继续等待用户通过点击发送或 Ctrl+Enter 主动发送，两者互不干扰。
- Q: 超链接 meta 解析是否仅限 HTTPS？ → A: 否。HTTP 和 HTTPS 均支持解析；安全级别（如是否允许 HTTP）后续通过应用设置项配置，规格阶段不限制协议。
- Q: 用户一次选择多个文件（含不同类型），如何处理？ → A: 每个文件各自识别类型（图片/视频/附件），创建对应类型的独立气泡，按文件顺序依次发送，不做类型限制。
- Q: 上传失败后点击重试，气泡如何处理？ → A: 原地替换：失败气泡状态重置为"上传中"，重试成功后显示内容，再次失败仍显示错误提示；气泡数量不变，不新增气泡。
- Q: FFmpeg 未安装导致视频缩略图生成失败时，气泡如何降级展示？ → A: 显示通用视频图标（播放按钮图案）+ 文件名，点击后在模态框中仍可正常播放视频。

## Assumptions

- 文件大小上限由服务端配置决定，规格阶段假定单文件上限为 100MB（可通过环境变量配置）
- 视频缩略图生成依赖服务器端工具（如 FFmpeg），如未安装则降级显示通用视频图标（播放按钮图案）+ 文件名，点击后仍可在模态框中正常播放
- 超链接 meta 解析在服务端进行，不依赖浏览器环境；HTTP 和 HTTPS 链接均默认支持解析，后续可通过应用设置项配置允许的协议范围
- 文本类附件内容预览仅支持 UTF-8 编码，超过一定大小（如 1MB）时截断显示
- 多媒体上传进度通过 SignalR 实时推送给客户端
- 所有文件存储在本地文件系统（`LocalFileStorageService`），与现有 MVP 保持一致
- 置顶功能是单用户级别的设置（本应用为私有单用户/小组场景）
- 历史记录分页每次加载 20 条，向上滚动触发加载
- 视频播放使用浏览器原生 `<video>` 标签，不引入额外播放器库
