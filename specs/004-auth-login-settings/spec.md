# Feature Specification: 认证、登录与设置功能

**Feature Branch**: `004-auth-login-settings`  
**Created**: 2026-04-19  
**Status**: Draft  
**Input**: 系统需要实现认证和登录功能，保证信息不泄露；支持首次使用时自动进入配置向导创建用户；未登录用户无法访问主页和共享内容；API 实现 JWT 认证；单用户体系（主要针对局域网）；设置页面支持密码/昵称修改和安全设置（是否自动获取 HTTP 链接 meta）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 首次配置向导 (Priority: P1)

第一次访问 AnyDrop 实例时，系统检测到尚无任何用户账号，自动将用户引导至「初始配置向导」。向导收集管理员昵称和密码，创建唯一用户账号，并在创建成功后自动以该账号登录，跳转至主页。

**Why this priority**: 没有首次配置流程，任何用户都无法开始使用系统，是整个认证体系的前置条件。

**Independent Test**: 在全新数据库实例中访问任意页面 → 被重定向至配置向导 → 填写昵称与密码 → 提交后自动登录并进入主页。

**Acceptance Scenarios**:

1. **Given** 数据库中没有任何用户账号，**When** 用户访问主页 `/`，**Then** 系统自动跳转到配置向导页面 `/setup`。
2. **Given** 用户在配置向导填写有效昵称和密码（两次密码一致），**When** 提交表单，**Then** 账号创建成功，自动登录，重定向至主页。
3. **Given** 用户填写的两次密码不一致或密码长度不足 8 个字符，**When** 提交表单，**Then** 显示具体错误提示，表单不提交，账号不创建。
4. **Given** 用户账号已存在，**When** 直接访问 `/setup`，**Then** 系统重定向至登录页或主页（向导不可再次访问）。

---

### User Story 2 - 登录与登出 (Priority: P1)

已有账号的用户通过登录页面输入密码进行身份验证。验证成功后获得有效凭证，访问受保护内容。可随时登出，登出后凭证立即失效。

**Why this priority**: 认证功能的核心入口，无此功能其他安全措施无从发挥。

**Independent Test**: 在已有用户的实例中，访问登录页 → 输入正确密码 → 进入主页；访问登录页 → 输入错误密码 → 显示错误，停留在登录页；登录后点击登出 → 返回登录页。

**Acceptance Scenarios**:

1. **Given** 用户在登录页输入正确密码，**When** 提交登录表单，**Then** 验证成功，获得访问凭证，跳转至主页（或登录前访问的目标页面）。
2. **Given** 用户输入错误密码，**When** 提交登录表单，**Then** 显示「密码错误」提示，停留在登录页，不透露账号是否存在。
3. **Given** 用户已登录，**When** 点击登出按钮，**Then** 凭证立即失效，跳转至登录页，后退键无法返回受保护页面。
4. **Given** 用户已登录并访问登录页 `/login`，**When** 页面加载，**Then** 自动重定向至主页（已登录用户无法访问登录页）。

---

### User Story 3 - 访问控制与页面保护 (Priority: P1)

所有受保护页面（主页、历史记录等）要求有效的登录凭证；未持有凭证的请求自动重定向至登录页面，并在登录后回到原目标页。API 端点要求 JWT Bearer Token，无效或缺失时返回 401 状态码。

**Why this priority**: 是「信息保密」目标的直接体现，确保未授权方无法访问任何共享内容。

**Independent Test**: 在未登录状态访问 `/`（主页）→ 被重定向至 `/login?returnUrl=/`；使用空 Authorization 头调用 `GET /api/v1/topics` → 响应 `401 Unauthorized`。

**Acceptance Scenarios**:

1. **Given** 用户未登录，**When** 访问主页或任何受保护页面，**Then** 自动重定向至 `/login?returnUrl=<原目标路径>`。
2. **Given** 用户未登录，**When** 调用任意受保护 API 端点（无 Authorization 头），**Then** 返回 `401 Unauthorized`（不返回数据）。
3. **Given** 用户使用过期或篡改的 JWT Token 调用 API，**When** 请求被处理，**Then** 返回 `401 Unauthorized`。
4. **Given** 用户登录后被重定向至 `/login?returnUrl=/share/xyz`，**When** 登录成功，**Then** 跳转至原目标页面 `/share/xyz`。

### User Story 4 - 设置页面：密码与昵称修改 (Priority: P2)

已登录用户通过系统内的「设置」页面修改自己的显示昵称和账号密码，修改立即生效，无需重新登录（密码修改后当前会话继续有效）。

**Why this priority**: 增强账号管理能力，但不影响核心共享功能，可在 MVP 后交付。

**Independent Test**: 进入「设置」→「账号」→ 修改昵称为「新昵称」→ 保存 → 刷新页面后昵称显示为「新昵称」；修改密码 → 保存 → 登出 → 用新密码登录成功。

**Acceptance Scenarios**:

1. **Given** 用户在设置页输入新昵称（1–50 字符），**When** 保存，**Then** 昵称更新立即反映在界面中，无需重新登录。
2. **Given** 用户输入当前密码（正确）和新密码（符合规则，两次一致），**When** 保存密码，**Then** 密码修改成功，当前会话继续有效，旧密码无法再登录。
3. **Given** 用户输入当前密码错误，**When** 尝试修改密码，**Then** 显示「当前密码错误」，密码不更改。
4. **Given** 用户输入新密码长度不足 8 个字符，**When** 尝试保存，**Then** 表单校验失败，显示错误提示，密码不更改。

### User Story 5 - 设置页面：安全设置 (Priority: P2)

管理员可在设置页「安全」选项卡中开启或关闭「自动获取 HTTP 链接 meta 信息」功能。此开关影响全系统（即 003 功能中的链接预览行为），修改后立即生效，无需重启。

**Why this priority**: 该设置直接影响隐私（抓取外部 URL 会产生出站请求），对安全敏感的用户有价值，但不阻塞核心功能。

**Independent Test**: 进入「设置」→「安全」→ 将「自动获取链接预览」关闭 → 保存 → 发送含 URL 的消息 → 不出现链接预览卡片。

**Acceptance Scenarios**:

1. **Given** 「自动获取链接预览」已开启，**When** 发送含 HTTPS URL 的消息，**Then** 系统异步抓取链接 meta 并显示预览卡片（现有行为）。
2. **Given** 「自动获取链接预览」已关闭，**When** 发送含 URL 的消息，**Then** 系统不进行任何出站请求，不显示链接预览卡片。
3. **Given** 管理员修改安全设置，**When** 保存，**Then** 设置立即生效，无需重启应用；同时刷新页面后设置值持久化。

### Edge Cases

- **首次配置向导被并发提交两次**：若两个请求同时尝试创建用户，系统应只接受第一个请求，第二个请求失败并返回「系统已完成初始配置」错误。
- **Token 在活跃会话中过期**：Blazor 页面上的 JWT 过期后，下一次页面交互或路由导航时重定向至登录页，避免无限制访问。
- **已有账号的情况下访问 `/setup`**：立即重定向至 `/login`，不显示向导内容，不允许二次配置。
- **密码修改后，浏览器仍持有旧 Cookie**：修改密码不强制使当前 session 失效，但登出后旧密码无法重新登录。
- **昵称为空或超长**：最小 1 个字符，最大 50 个字符，超出范围显示校验错误，不保存。
- **JWT Secret 未配置**：系统启动时检测到 JWT 配置缺失，在日志中输出警告并拒绝启动（fail-fast 策略）。

## Requirements *(mandatory)*

### Functional Requirements

**认证核心**

- **FR-001**: 系统 MUST 在没有任何用户账号时，将所有页面（`/setup` 除外）的请求重定向至初始配置向导 `/setup`。
- **FR-002**: 配置向导 MUST 收集昵称（必填，1–50 字符）和密码（必填，最少 8 字符，两次输入一致），创建单一系统账号。
- **FR-003**: 系统 MUST 仅允许存在一个用户账号；第二次尝试创建账号 MUST 被拒绝。
- **FR-004**: 系统 MUST 提供登录页面 `/login`，接受密码输入并进行身份验证。
- **FR-005**: 登录成功后，系统 MUST 向客户端签发有效的访问凭证（有效期默认 24 小时，可通过配置调整）。
- **FR-006**: 密码 MUST 以不可逆哈希（含 salt）形式存储，任何情况下不得以明文保存或传输。
- **FR-007**: 已登录用户访问登录页 `/login` 或配置向导 `/setup` 时，MUST 自动重定向至主页 `/`。

**访问控制**

- **FR-008**: 所有受保护的 Blazor 页面 MUST 要求有效的登录凭证；未认证请求 MUST 重定向至 `/login?returnUrl=<原始路径>`。
- **FR-009**: 所有受保护的 Minimal API 端点 MUST 要求有效的 JWT Bearer Token；无效或缺失时 MUST 返回 `401 Unauthorized`，不附带业务数据。
- **FR-010**: JWT Token 中 MUST 包含用户标识信息，API 层可据此识别请求方。
- **FR-011**: 登出操作 MUST 立即使当前凭证失效，使后续对受保护资源的访问跳转至登录页；系统 MUST 通过会话版本号机制（JWT 携带 SessionVersion，服务端验证当前版本）实现已签发令牌即时吊销。

**设置功能**

- **FR-012**: 已认证用户 MUST 能从设置页修改自己的昵称（1–50 字符）。
- **FR-013**: 已认证用户 MUST 能修改密码，修改前 MUST 验证当前密码正确；新密码须符合最少 8 字符规则且两次输入一致。
- **FR-014**: 密码修改成功后，当前会话 MUST 继续有效（不强制重新登录）。
- **FR-015**: 系统 MUST 提供一个全局布尔开关「自动获取链接预览」，持久化存储在数据库中。
- **FR-016**: 「自动获取链接预览」开关的值 MUST 在系统启动时加载，并在值改变后即时生效（无需重启）。

**安全基线**

- **FR-017**: 登录接口 MUST 对连续失败尝试进行速率限制（5 次失败后触发冷却，避免暴力破解）。
- **FR-018**: JWT 签名密钥 MUST 从环境变量或配置文件读取，不得硬编码。
- **FR-019**: 错误的登录尝试 MUST 返回通用错误消息（「用户名或密码错误」），不揭示账号是否存在。
- **FR-020**: 系统 MUST 在登出与密码修改成功时递增会话版本号，确保旧 JWT 在下一次请求时被拒绝（401）。

### Key Entities

- **User**（用户账号）：系统内唯一用户实体。包含：用户标识、昵称（用于界面显示）、密码哈希（含 salt）、创建时间、最后登录时间。
- **SystemSettings**（系统设置）：单行全局配置表，存储安全及行为开关。初始字段：`AutoFetchLinkPreview`（布尔，默认 `true`）。可按需扩展更多设置项。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在全新实例上，用户完成首次配置向导并进入主页的全过程不超过 2 分钟。
- **SC-002**: 登录成功后，主页加载完成（可交互状态）不超过 3 秒（局域网环境）。
- **SC-003**: 错误密码输入后，错误提示在 500ms 内显示，无页面重载。
- **SC-004**: 访问受保护页面时，未认证状态下的重定向在 300ms 内完成。
- **SC-005**: API 对无效或缺失 JWT Token 的请求，401 响应在 100ms 内返回。
- **SC-006**: 昵称修改保存后，界面中的显示名称在 1 秒内更新，无需刷新页面。
- **SC-007**: 「自动获取链接预览」开关关闭后，发送含 URL 的消息，系统不发出任何出站 HTTP 请求（可通过网络日志验证）。
- **SC-008**: 密码和设置变更 MUST 在应用重启后仍持久有效（数据库持久化）。

## Assumptions *(记录合理默认值)*

- **单用户体系**：系统仅支持一个用户账号，无多用户、角色、权限概念。
- **会话时长**：访问凭证有效期默认 24 小时，可通过 `Auth:TokenExpiryHours` 配置项调整。
- **令牌吊销策略**：采用短时 JWT + 会话版本号（SessionVersion）方案，不使用黑名单存储。
- **会话时长**：访问凭证有效期默认 24 小时，可通过 `Auth:TokenExpiryHours` 配置项调整。
- **登录标识**：由于只有单用户，登录表单只需要密码字段（不需要用户名输入）；内部使用固定标识区分账号。
- **Rate Limiting 策略**：5 次失败 → 60 秒冷却，不永久锁定账号（局域网环境避免自锁）。
- **密码规则**：最少 8 个字符，无其他强制复杂度要求（可后续在安全设置中扩展）。
- **Blazor 认证机制**：使用 Cookie 认证（`CookieAuthenticationHandler`）管理 Blazor Server 页面会话；Minimal API 使用 JWT Bearer Token 认证，两者独立配置，互不干扰。
- **配置向导 URL**：`/setup`，不包含在导航菜单中。
- **设置页 URL**：`/settings`，通过现有导航中的设置图标/按钮进入。
- **SystemSettings 初始化**：应用首次启动时若 `SystemSettings` 表为空，自动插入默认行（`AutoFetchLinkPreview=true`）。
